## 1 项目目标

### 1.1 产品要解决什么问题

**目标用户**：客户经理、数据分析岗等。

**核心痛点**：
- 用户想问一个业务问题，但报表多，数据分散在多个报表，查一个答案要开N个文件。
- 整合难，从不同报表分别取数后，还需手动关联、计算，才能得到完整结果。
- 依赖SQL，业务人员无法自助查询。

**产品价值**：用户通过自然语言提问，系统自动把“业务问题”变成“结构化查询”，并返回“可信结果”。

### 1.2 三个核心能力

| 能力 | 说明 |
|:---|:---|
| 理解业务问题 | 用户用日常语言即可提问，系统自动将自然语言解析为意图、实体、指标、参数 |
| 拆解为结构化查询 | 用户无需知道数据在哪张表、如何查询，系统自动将解析结果转化为SQL |
| 获得可信结果 | 系统返回准确数据 |

#### 2.1.5 提示词管理

在实际使用中，用户问法多样，同一个业务概念可能有多种表达方式。例如：
- "风险等级"也可能被说成"风险评级"、"风险级别"
- "持仓规模"也可能被说成"持有金额"、"持仓市值"

如果只认识标准词，大量用户问法将无法识别，导致：
- 识别准确率低，用户体验差
- 需要频繁修改代码来适配新问法，维护成本高
- 业务方无法自主扩展词汇，依赖开发排期

构建提示词，将业务术语、同义词、映射关系、数据类型、口径定义等进行结构化管理和可视化维护，提升识别准确率，支持业务方自主扩展词汇。

| 分类 | 说明 | 示例 |
|:---|:---|:---|
| 场景 | 业务域关键词 | 客户、持仓、交易、产品 |
| 实体 | 业务对象同义词 | 客户→委托人、投资者 |
| 指标 | 指标同义词 | 风险等级→风险评级、风险级别 |
| 参数 | 条件同义词 | 大于→超过、高于 |

### 2.2 第二步：拆解为结构化查询

#### 2.2.1 拆解的核心逻辑

将自然语言解析结果转化为可执行的SQL。

**拆解公式**：业务问题 = 查询对象 + 筛选条件 + 输出字段 + 计算逻辑 + 输出形式。

**拆解示例**：

| 用户问法 | 查询对象 | 筛选条件 | 输出字段 | 计算逻辑 | 输出形式 |
|:---|:---|:---|:---|:---|:---|
| 李丽的风险等级是什么 | 客户 | 客户名称等于李丽 | 风险等级 | 无 | 卡片 |
| 持仓超过1000万的客户有哪些 | 客户 | 资产总规模大于1000万 | 客户名称、资产总规模 | 无 | 表格 |
| 本月认购总金额是多少 | 交易 | 交易类型为认购，月份为本月 | 认购金额 | 求和 | 卡片 |
| 李丽的持仓收益率是多少 | 持仓 | 客户名称等于李丽 | 收益率 | (市值减成本)除以成本 | 卡片加说明 |

#### 2.2.2 拆解步骤

**第一步：识别查询对象**

| 用户表述 | 查询对象 | 对应宽表 |
|:---|:---|:---|
| 客户、委托人、投资者、持有人 | 客户 | 客户宽表 |
| 持仓、持有、买了什么 | 持仓 | 持仓宽表 |
| 交易、申购、赎回、流水 | 交易 | 交易宽表 |
| 收益率、占比、集中度 | 计算指标 | 多表关联或计算 |

**第二步：识别筛选条件**

| 条件类型 | 识别方式 | 示例 |
|:---|:---|:---|
| 等值条件 | "是"、"为"、"叫" | "客户是李丽"转为客户名称等于李丽 |
| 比较条件 | "大于"、"小于"、"超过"、"低于" | "超过1000万"转为金额大于1000万 |
| 范围条件 | "之间"、"从...到..." | "3月1日到3月31日"转为日期介于两个日期之间 |
| 集合条件 | "在...中"、"包含" | "C5客户"转为风险等级在C5中 |
| 模糊条件 | "像"、"含有" | "名字带'李'"转为客户名称包含李字 |
| 否定条件 | "不是"、"除外" | "不是机构客户"转为客户类型不等于机构 |

**第三步：识别输出字段**

| 用户表述 | 输出字段 | 说明 |
|:---|:---|:---|
| "客户名称" | 客户名称 | 明确指定 |
| "风险等级是什么" | 风险等级 | 单个属性 |
| "客户有哪些" | 默认展示字段 | 使用预定义的默认字段列表 |
| "持仓明细" | 默认展示字段 | 使用预定义的默认字段列表 |
| "全部信息" | 默认展示字段 | 使用预定义的默认字段列表 |

**第四步：识别计算逻辑**

| 计算类型 | 用户表述 | 计算公式 | 是否需要聚合 |
|:---|:---|:---|:---|
| 汇总 | 总金额、合计、总共 | 对字段求和 | 是 |
| 计数 | 多少个、有多少 | 统计记录数 | 是 |
| 平均 | 平均、人均 | 对字段求平均值 | 是 |
| 最大/最小 | 最大、最小、最高、最低 | 取字段最大值或最小值 | 是 |
| 占比 | 占比、比例、百分比 | A除以B再乘以100% | 是 |
| 收益率 | 收益率、回报率、赚了多少 | (当前值减成本)除以成本 | 是 |
| 净流入 | 净流入、净申购 | 申购总额减赎回总额 | 是 |
| 环比/同比 | 环比、比上月、同比、比去年 | (本期减上期)除以上期 | 是 |

**第五步：识别输出形式**

| 数据特征 | 自动选择的展示形式 |
|:---|:---|
| 单行单列 | 卡片 |
| 多行多列，行数不超过20 | 表格 |
| 多行多列，行数超过20 | 表格加分页，每页20行 |
| 2至6个分类的计数 | 饼图 |
| 7个以上分类的计数 | 柱状图 |
| 带时间序列的数据 | 折线图 |
| 有排序列且要前N名 | 水平条形图 |

#### 2.2.3 拆解结果的结构化输出

用户问一句话，拆解后应该输出一个结构化的查询配置，包含以下信息：查询类型、查询对象、筛选条件列表、输出字段列表、计算逻辑、输出形式、解释说明。

### 2.3 第三步：获得可信结果

#### 2.3.1 数据可信的四个维度

| 维度 | 说明 |
|:---|:---|
| 准确性 | 定义明确的口径规则，通过QA对验证 |
| 完整性 | 制定空值处理规则和默认值设置 |
| 时效性 | 标注数据日期，采用T-1更新策略 |
| 可解释性 | 展示思考过程和口径说明 |

#### 2.3.2 结果呈现策略

| 顺序 | 输出结果 | 判断条件 | 展现形式 | 分页规则 |
|:---:|:---|:---|:---|:---|
| 1 | 1个字段 | 返回结果只有1个字段 | 文字卡片 | 无 |
| 2 | 2个字段 | 第一个字段为日期型，第二个字段为数值型 | 折线图 | 无 |
| 3 | 2个字段 | 第一个字段为文本型，第二个字段为数值型，返回行数不超过12 | 柱状图 | 无 |
| 4 | 2个字段 | 两个字段均为数值型，或不符合以上3种情形 | 表格 | 行数小于20全展示，大于等于20分页（每页20行） |
| 5 | 超过2个字段 | 返回3个及以上字段 | 表格 | 行数小于20全展示，大于等于20分页（每页20行） |

#### 2.3.3 敏感数据脱敏处理

**脱敏原则**：
- 脱敏与权限无关：无论用户权限高低，敏感字段统一脱敏展示
- 脱敏发生在展示层：后端返回原始数据，前端统一脱敏
- 导出与展示一致：导出时同样执行脱敏

|:---|:---|:---|:---|
| 客户姓名（2字） | 保留姓氏，名字用星号代替 | 张三 | 张* |
| 客户姓名（3字及以上） | 保留首字，其余用星号代替 | 张美美 | 张** |
| 身份证号 | 保留前4位，中间用星号，保留后4位 | 41010119900307663X | 4101********7663X |

#### 3.6.5 SQL层权限注入

权限在SQL生成后由系统层注入，不在提示词中体现，避免大模型修改。

**普通用户查询注入规则**：

系统自动在查询条件中增加客户归属过滤，限定当前用户只能查询其管户关系表中记录的客户。过滤条件为：客户ID必须在用户管户关系表中，且该关系未过期。

**团队管理者查询注入规则**：

系统自动在查询条件中增加团队归属过滤，限定当前用户只能查询其所属团队下的客户。过滤条件为：客户所属团队ID等于当前用户所在的团队ID。

**数据分析岗及系统管理员**：

无额外过滤条件，可查询全量数据。

#### 3.7.1 模块定位

记录所有用户查询行为和系统操作，支持审计、问题排查和模型优化。**特别记录所有问答（含兜底回答）**，用于迭代和优化问答模型，持续提升准确性。

#### 3.7.2 核心价值

| 价值 | 说明 |
|:---|:---|
| 模型优化 | 记录未识别问题和兜底回答，分析原因，补充提示词 |
| 准确性提升 | 通过日志分析，持续提升场景识别、实体识别的准确率 |
| 问题追踪 | 追溯每次问答的完整链路，定位问题根因 |
| 合规审计 | 满足数据安全合规要求，可追溯所有查询行为 |

#### 3.7.3 功能列表

| 功能 | 说明 | 优先级 |
|:---|:---|:---|
| 查询日志记录 | 自动记录每次用户提问（含兜底回答） | P0 |
| 兜底回答标记 | 标记该次回答是否为兜底回答及具体类型 | P0 |
| 未识别问题分析 | 统计未识别问题，供提示词补充 | P1 |
| 准确率统计 | 按场景或时间统计识别准确率 | P1 |
| 日志检索 | 支持按时间、用户、问题关键词、兜底类型检索 | P1 |
| 日志导出 | 导出日志为Excel或CSV格式 | P2 |

#### 3.7.4 日志记录内容

日志记录包含以下字段：日志ID、用户ID、用户角色、操作时间（精确到毫秒）、查询问题、识别场景、识别实体、是否兜底回答、兜底回答类型（A/B/C/D/E/F）、生成的SQL、返回数据量、响应时间、是否成功、失败原因、用户反馈（有用/无用/未反馈）、客户端IP。

#### 3.7.5 兜底回答专项记录

所有兜底回答**必须记录**以下额外字段，用于模型迭代：

| 字段 | 说明 |
|:---|:---|
| 兜底类型 | A/B/C/D/E/F |
| 触发原因 | 具体原因（如：非问数问题、图表不可用、查询无结果等） |
| 原始问题 | 用户原始输入 |
| 建议补充提示词 | 系统自动分析建议（如适用） |

#### 3.7.6 未识别问题分析

从日志中提取系统无法识别的问题（未命中任何场景或实体、触发兜底回答），按出现频次排序，供提示词补充使用。

**分析维度包括**：
- 未识别问题TOP20（按出现次数排序）
- 未识别问题趋势（按天展示未识别率变化）
- 建议新增同义词（基于未识别问题中的高频词汇）
- 兜底回答分布（各类兜底回答的占比和趋势）
- 场景准确率（各业务场景的识别准确率）

#### 3.7.7 数据存储

| 配置项 | 说明 |
|:---|:---|
| 存储方式 | 数据库加对象存储（冷热分离） |
| 热数据保留 | 30天（存储于数据库） |
| 冷数据保留 | 180天（存储于对象存储） |
| 清理策略 | 超过180天的日志自动归档删除 |

### 3.8 兜底回答

#### 3.8.1 模块定位

当系统无法正常响应用户请求时，给出明确、友好的兜底回答。**所有兜底回答记录到日志，用于模型优化迭代**。

#### 3.8.2 兜底回答类型

| 类型 | 触发条件 | 回答模板 | 迭代用途 |
|:---|:---|:---|:---|
| A类：非问数问题 | 与数据查询无关（如闲聊、问候） | 表明自己是信托数据查询助手，请用户提出数据查询问题，并给出示例 | 分析用户真实意图，考虑扩展场景 |
| B类：非场景问题 | 不在四大业务场景内 | 说明目前只支持客户、持仓、交易、产品四类业务场景 | 评估是否需要新增业务场景 |
| C类：图表不可用 | 用户要求图表但数据特征不满足 | 说明无法按用户要求展示图表的原因（如饼图最多支持6个分类），已自动切换为表格展示 | 分析图表需求，优化展示规则 |
| D类：查询无结果 | 查询条件正确但无匹配数据 | 提示未找到符合条件的数据，建议放宽筛选条件或扩大时间范围 | 分析是否为数据缺失问题 |
| E类：权限不足 | 用户无权限查询该数据 | 提示用户暂无权限查看该数据，如需开通请联系管理员 | 审计权限需求，评估是否需要调整权限策略 |
| F类：系统异常 | 超时或服务不可用 | 提示系统暂时无法完成查询，请稍后重试或联系管理员 | 监控系统稳定性 |

#### 3.8.3 迭代闭环

用户提问后，系统处理，如触发兜底回答则记录日志，定期分析日志，根据分析结果优化模型或补充提示词，从而减少兜底回答的发生。
